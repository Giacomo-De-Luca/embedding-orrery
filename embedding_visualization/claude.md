# CLAUDE.md - Embedding Visualization Frontend

Next.js 15 web app for visualizing embedding collections with interactive 2D/3D scatter plots and semantic search. Supports flexible metadata with automatic field detection.

## Development

```bash
npm run dev                    # Frontend only (http://localhost:3000)
npx shadcn@latest add [name]  # Add UI components
```

## Component Architecture

```
page.tsx (orchestration)
  ├─ AppHeader (collection selector, semantic search, theme toggle)
  ├─ DashboardPanel (resizable layout via react-resizable-panels)
  │   ├─ ScatterPlot2D (Plotly WebGL)
  │   ├─ ScatterPlot3D (Plotly 3D with spherical camera interpolation)
  │   ├─ Legend (category colors, gradient legend, click-to-mute)
  │   └─ SimilarItemsTable (semantic search results, @tanstack/react-table)
  └─ EmbeddingSidebar (controls + selected point info + text search)
```

**Collections page** (`/collections`, formerly `/test-embed`): Three tabs — HuggingFaceTab, LocalFileTab, CollectionManagerTab (with TopicExtractionCard). Each tab manages own state; shared `useEmbedDataset` hook for GraphQL operations. **ProgressModal** (`app/collections/components/EmbeddingProgressModal.tsx`): Reusable modal with WebSocket subscription, elapsed timer, ETA calculation, configurable `title`/`subtitle`/`itemsLabel` props. Used for embedding progress, topic extraction progress, topic reduction progress, LLM labeling progress (all in TopicExtractionCard), and per-projection progress. ETA shown after 2+ progress data points as "~X:XX remaining". JobsPanel handles `llm_labeling` job type alongside embedding jobs. **TopicConfigForm** (`app/collections/components/TopicConfigForm.tsx`, shared by TopicExtractionCard, SaeTab, and the embed-with-topics flow): exposes one merged **"Clustering Space" select** instead of separate Projection Type / Clustering Space controls (the projection toggle was ignored whenever a raw-vector mode was active). State field `clusteringSpace` holds either a stored projection (`umap_3d` default, `umap_2d`, `pca_3d`, `pca_2d` → sent as `clusterOn:"projection"` + that `projectionType`) or a raw-vector mode (`cluster_umap`/`embedding` → sent as `clusterOn`, with `projectionType` pinned to `umap_3d` for the backend's id/document load + fallback). `toTopicConfigInput` does the split back to the GraphQL pair.

## State Management

Visualization state is managed by a **Zustand store** (`lib/stores/useVisualizationStore.ts`) with `subscribeWithSelector` middleware. Components read state via selectors for granular re-renders — no prop drilling of visualization state.

- **Store shape**: `VisualizationStoreState` defines all fields; `VisualizationStoreActions` defines mutations
- **ColorScale union** (`lib/types/types.ts`): Discriminated union `{ type: 'categorical' } | { type: 'sequential'; scaleName } | { type: 'diverging'; scaleName } | { type: 'monochrome'; baseColor }`. Replaces the old flat fields (`colorScaleType`, `sequentialScaleName`, `divergingScaleName`, `monochromeColor`). `categoricalPalette` stays separate — chart components need it regardless of scale type.
- **Auto-reset subscription**: Store subscribes to `colorByField` changes and auto-clears `mutedCategories` + `hideUnclustered` (synchronous, no React effect needed)
- **Self-contained controls**: `ColorScaleSelector` and `VisualizationControls` read/write the store directly with zero props for state
- **Data stays as props**: Points, search results, topics, and other computed data flow via props (not the store) since they come from GraphQL/hooks

## Key Patterns

- **Self-contained components**: Each manages its own loading/error/success states
- **Progressive loading**: App shell → collections → data (no full-page blocking)
- **Auto-detection** (`lib/utils/fieldAnalysis.ts`): Label fields (word, title, name, label), category fields (pos, category, type, topic), numeric vs categorical detection
- **Color system** (`lib/utils/categoryColors.ts`): Categorical (D3 + Crameri palettes), sequential (including `rainbow` via `interpolateRainbow`), diverging, monochrome scales. Presets for known types (POS, Topic). Crameri colormaps lazy-loaded from `lib/colorMaps/`. Custom color strips (hilbertColor, hueSatColor, xkcdColor) generated by backend script, stored as Crameri-format JSONs. Backend preprocessing maps hex color columns to `mapped_colour` float (0-1) for use with these strips. **Nested color mode** (`lib/utils/nestedColorUtils.ts` + `lib/hooks/useNestedCategoryData.ts`): Tableau-style hierarchical coloring where topics define base hues and subtopics get lightness variations. Activated via `nestedColorMode` toggle when `topic_label` is selected and `subtopic_label` exists.
- **Color scheme persistence** (`lib/utils/colorScaleUrl.ts`): The active colouring — field (`colorBy`), the `ColorScale` union (flattened to `scale`/`scaleName`/`color`) and `categoricalPalette` — round-trips through the URL query string (read/written in `page.tsx`), so shared links reproduce the exact look. Each collection can also store a **default colour scheme** in `extra_metadata.default_color_scheme` (JSON string, written by `SaveColorDefaultButton` via `updateCollectionMetadata`, parsed in `useCollections` → `CollectionInfo.defaultColorScheme`). On collection load `page.tsx` applies it with precedence **URL > collection default > none**; a per-collection `defaultsAppliedFor` ref plus `useEmbeddingData`'s returned `loadedCollection` gate the apply effect so it never clobbers manual edits or applies against the previous collection's stale `colorFieldOptions`. Shared helpers: `serializeColorScale`/`deserializeColorScale` (URL params) and `serializeDefaultColorScheme`/`resolveDefaultColorScheme` (per-collection default). No backend changes — relies on `extra_metadata` merge/spread.
- **Nebula cluster effects** (`lib/utils/clusterGeometry.ts` + `lib/utils/hazeRenderer.ts`): Boolean toggle (`nebulaMode`) adds translucent haze sprites around topic clusters. Uses `HazeRenderer` on a separate overlay canvas with `mix-blend-mode: screen`, hooked into Plotly's GL render loop for camera-synced projection. Deprecated renderers (volume isosurface, WebGL sprites, Three.js bloom) archived in `lib/utils/experimental/`. **Per-frame cost is minimized** in `ScatterPlot3D`: the renderer lifecycle (create/dispose + `glplot.onrender` hook, keyed on `nebulaMode`/`plotReady`/`isDark`/has-clusters) is split from cluster-content updates, so muting/filtering/recoloring only calls `updateClusters` on the existing renderer instead of re-instantiating the Three.js `WebGLRenderer`; the overlay-canvas sync writes styles + backing-store dims only on an actual offset/size change (`lib/utils/overlayLayout.ts` — sharing that math with the label overlay, avoiding a per-frame `canvas.width` reset + reflow); and the data→scene model matrix is cached per `bounds` identity (pinned axis ranges make it constant during camera interaction). Each haze sprite is still its own draw call — batching to `THREE.Points` (one draw call per cluster) is the documented Tier 2 follow-up. `clusterDataMap` excludes points muted by text-search/temporal filters (`combinedMutedIndices`) in addition to `mutedCategories`/`hideUnclustered`, so haze and cluster labels track visibly active points; `HazeRenderer.dispose()` clears the drawing buffer to the neutral blend color so no stale haze frame lingers on the still-mounted overlay canvas when filtering empties all clusters.
- **Label collision avoidance** (`app/utils/labelPlacement.ts` + `lib/utils/collisionGrid.ts`): On each rAF-polled camera change, projects 3D label positions to 2D screen space via MVP matrices from Plotly's WebGL internals, measures text bounding boxes on a canvas overlay, then uses `CollisionGrid` with two-pass insertion (thin probe + full box) for greedy set packing. Higher-similarity labels survive collisions; selected point gets infinite priority. **Cluster labels** (`showClusterLabels` on `VisualizationState`): displays topic/subtopic names at cluster centroids (bold 13px, cluster color) with highest collision priority, sharing the same `CollisionGrid` as point labels.
- **Resizable Legend** (`lib/hooks/useVerticalResize.ts`): Legend card has a drag handle at its bottom edge controlling `maxHeight` via mouse/touch events. Dragging below the collapse threshold (60px) collapses to a pill button (Palette icon); clicking the pill re-expands at default height. Uses callback ref pattern for conditionally rendered elements.
- **Screenshot export** (`lib/utils/plotCapture.ts`): A camera modebar button (rightmost, both plots) downloads one PNG of the current visualization — plot + overlay canvases + legend, no UI — at screen size × devicePixelRatio. Background is filled explicitly (pure black in dark mode, white in light) because the plot/paper backgrounds are transparent in dark mode. 2D captures via `Plotly.toImage` (scattergl axes are SVG, a live-canvas copy would miss them) then draws the label canvas on top; 3D reads the live GL scene via the fork's `scene._scene.toImage()` (synchronous redraw + readPixels — immune to `preserveDrawingBuffer:false` and to the fly-to camera lagging the layout) and composites the haze canvas with `globalCompositeOperation` matching its CSS `mix-blend-mode`, then labels. The plots' Plotly `config` memos only rebuild on plotly-ready, so the button's click handler goes through a `screenshotHandlerRef` reassigned every render (same anti-staleness pattern as `renderLabelsRef`). Each plot composites what it owns and calls its `onScreenshot(canvas)` prop; `DashboardPanel` then rasterizes the **live legend DOM node** with `html-to-image` (`drawLegendOverlay` — the real component, histogram and all; skipped when legend hidden/collapsed since the overlay ref is unmounted; elements marked `data-export-exclude`, e.g. the drag handle, are omitted) over a canvas-blurred backdrop replicating the Card's `backdrop-blur-sm` (foreignObject rasterization can't reproduce backdrop-filter), and downloads via `downloadCanvasPng` (`downloadJson.ts`).
- **Responsive plots**: `useContainerDimensions` (ResizeObserver) + Plotly `uirevision: 'true'` preserves zoom/pan
- **Search flow**: Text search is **server-side** via the `textSearch` GraphQL query (`lib/hooks/useTextSearch.ts`). Backed by DuckDB `ILIKE` substring scans on the per-dataset items table (`json_extract_string` for metadata fields) — see `duckdb_client.text_search()`. `SearchSidebar` Advanced section has field selector (multi-select: Document + metadata fields), match mode (Contains/Exact), and case-sensitivity toggle. Default: document-only, contains, case-insensitive. `TextSearchConfig` state stored in Zustand (`useVisualizationStore`). Text search uses **muting** (not glow) — non-matching points turn gray/low-opacity while matches keep normal colors. Text search muting (`searchMutedIndices`) combines with temporal muting (`temporallyMutedIndices`) into `combinedMutedIndices` in both scatter plots. Clicking a text search result in the sidebar triggers semantic search via GraphQL → similar items get glow overlay on top of muting. `HighlightedText` component marks matching text in both labels and documents in search results. **Topic search** (`lib/hooks/useTopicSearch.ts` + `app/components/TopicSearchSection.tsx`): Browse/filter topics by name/keywords, or rank topics by semantic similarity. Selecting topics highlights their points and scopes semantic search via backend `filters` parameter.
- **Synchronized topic selection**: When `colorByField === 'topic_label'`, `selectedTopicIds` (from `useTopicSearch`) is the single source of truth. `DashboardPanel` derives `effectiveMutedCategories` via `useMemo` (no state writes, single render cycle). Legend clicks, TopicSearchSection toggles, and 3D cluster label clicks all call `toggleTopic()`. Non-topic color fields preserve existing `mutedCategories` behavior. 3D cluster labels are clickable via `onClickCapture` hit-testing against bounding boxes stored during `renderLabels()`.
- **Filtered point controls**: `hideFilteredPoints` (boolean) on `VisualizationState` removes muted points entirely from the plot. `mutedPointOpacity` (0–1, default 0.20) is a **multiplier** applied to the current base opacity (not an absolute value) — muted opacity = base opacity × multiplier. This scales properly with point count since base opacity already decreases for large datasets. Toggle + slider UI in VisualizationControls. Reset on collection change.
- **Zoom-out limit** (`lib/hooks/useZoomLimit.ts`): Shared hook intercepting wheel events in capture phase before Plotly processes them. 3D reads live camera distance from `glplot.camera.eye` (max 3.0); 2D reads axis ranges from `_fullLayout` (max 2x data extent). Zoom-in, pan, rotation unaffected.
- **Deferred selected point** (3D only): On point click, `selectedPoint` drives camera fly-to via direct glplot manipulation (no Plotly.react). Trace-visible `renderedSelectedPoint` only syncs when `highlightedIndices` (search results) arrives, keeping `plotData` stable during animation to avoid main-thread stalls on large datasets.
- **Analytics category list** (`app/components/charts/CategoryBarList.tsx` + `lib/utils/categoryRowData.ts`): Category distribution as horizontal background-fill rows — a translucent category-color layer spans each row's background (width ∝ count, normalized to the max visible category) with label + count (`count (pct%)`) on top. All categories render in a 320px scrollable list (no top-N cap). Independent field selector dropdown (categorical fields from `colorFieldOptions`), defaulting to the active color field. When a text-search or temporal filter is active (`filteredCounts` prop, computed from `combinedMutedIndices`), rows show a faded total track + solid matching fill and counts flip to "matches / total". Sort ToggleGroup: Count (default; matches-desc while filtered) / Rate (match rate, only offered while filtered) / Name (numeric-aware natural order via `sortPeriods`). Category name filter input appears when >15 categories, matching resolved display labels. Row/sort/summary logic is pure and unit-tested (`lib/utils/__tests__/categoryRowData.test.ts`). Click row = isolate, shift+click = toggle (same `onCategoryToggle` contract). Unclustered noise (HDBSCAN label -1) is excluded from the distribution for **both topic and subtopic** fields via `getUnclusteredValues()` (`categoryColors.ts`, noise tokens derived from `TOPIC_PRESET`) — chart-only, independent of the scatter `hideUnclustered` toggle. The superseded recharts `CategoryBarChart.tsx` is retained temporarily pending verification of the swap.
- **Temporal filtering** (`lib/utils/temporalAnalysis.ts` + `lib/hooks/useTemporalData.ts` + `app/components/charts/TemporalFilterChart.tsx`): Auto-detects year/date fields via name heuristics and numeric range analysis. Analytics sidebar shows temporal distribution chart even without a color field selected (standalone mode: simple count-per-period area chart; stacked mode: category breakdown when color field is set). Custom DOM overlay range picker with draggable handle bars and a slidable selected region lets users filter points by time period — points outside the selected range are muted (gray, low opacity) in both 2D and 3D scatter plots via trace-splitting. Double-click resets to full range. `startTransition` + `useDeferredValue` keeps interaction non-blocking. Temporal range persists across color field changes but clears on collection change. Syncs with topic muting (muted categories excluded from stacked chart).

## Troubleshooting

- **Plotly types**: Use type assertions (`as Partial<Config>`)
- **Color scale not loading / custom strip shows as viridis**: Crameri/strip colormaps (e.g. `hilbertColor`) are lazy-loaded JSON chunks read synchronously from cache via `getCrameriPlotlyScale()`; an unloaded strip falls back to viridis. The scatter plots call `useColorScaleReady(colorScale)` (`lib/hooks/useColorScaleReady.ts`) to lazy-load **the active** strip and bump a tick that forces the colorscale `useMemo` to recompute when it lands — needed because a scale set programmatically (URL param / collection default) never opens `ColorScaleSelector` (which is the only other thing that triggers the load). Include that tick in any new colorscale memo deps.
- **Categories all gray**: Check `buildCategoryColorMap()` — presets only override specific values
- **`scene.scatter2d.update is not a function` / 2D scattergl renders nothing (3D fine)**: two related causes. (1) plotly marks regl components with the placeholder `true` during the calc pass (`calc.js`) and only creates the real objects in the dirty plot call (`plot.js`); a pan/zoom drag or box-select landing in that window called `.update()` on the boolean. Patched in the fork (`forked/plotly.js/src/traces/scattergl/scene_update.js`, `select.js`, `edit_style.js`) with `isReady()` guards. (2) The regl-* packages call `glslify()` **at runtime** around their pre-compiled shader strings, so `next.config.ts` must NOT alias glslify to `false` — that makes `createScatter()` throw (silently swallowed by react-plotly, which has no `onError`), leaving `scene.scatter2d` stuck on the placeholder → blank plot. The alias points to `glslify-runtime-stub.js` (tagged-template join) instead. Note the breakage only surfaces after a clean `.next` rebuild — stale webpack-cached chunks can mask config-level regressions for a long time. Also: never run two `next dev` instances against the same `.next` (corrupts chunks → ENOENT / 404s / silent server death).
