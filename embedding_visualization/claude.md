# CLAUDE.md - Embedding Visualization Frontend

Next.js 16 web app for visualizing embedding collections with interactive 2D/3D scatter plots and semantic search. Supports flexible metadata with automatic field detection.

## Development

```bash
npm run dev                    # Frontend only (http://localhost:3000)
npx shadcn@latest add [name]  # Add UI components
```

## Build & bundler (Next.js 16)

Next.js 16 makes **Turbopack the default** for `next dev` and `next build`. Two build scripts use **different bundlers on purpose**:

- `npm run build` → `next build --turbopack` — local/default build.
- `npm run build:docker` → `next build --webpack` — Docker/production image. Opts back into **Webpack** because the `webpack()` function in `next.config.ts` is still needed there, and Next 16 **fails** a Turbopack build when a `webpack` config is present unless a bundler flag is passed. Also sets `ORRERY_DOCKER_BUILD=1` (flips `typescript.ignoreBuildErrors`).

**Forked-Plotly aliases — Turbopack ignores the `webpack()` function, so both must be declared in `turbopack.resolveAlias` too:**
- **glslify** → `./glslify-runtime-stub.js` (the SAME runtime stub webpack uses). Must be the runtime stub, **not** an empty module / `false` — the regl-* packages call `glslify()` at runtime around their pre-compiled shaders, so an empty module makes `createScatter()` throw (react-plotly swallows it) and **2D scattergl renders blank**. See the troubleshooting note below.
- **maplibre-gl.css** → `./maplibre-css-stub.js` (empty JS module). `forked/plotly.js/src/registry.js` `require()`s this CSS for unused map traces; Turbopack **cannot instantiate a CSS module required from JS** ("module factory is not available"), which aborts the entire Plotly load. Aliasing CSS→CSS does not work — the target must be a JS module. Webpack imports the real CSS fine, so it's Turbopack-only.

`reactCompiler: true` is a top-level option (promoted out of `experimental` in Next 16); it relies on Babel, which makes the Webpack `build:docker` much slower than the Turbopack build. Six pre-existing broken/abandoned files (unimported, importing uninstalled `@uwdata/mosaic-*` etc.) are in `tsconfig.json` `exclude` so `next build`'s whole-program type-check passes — delete them as follow-up.

## Component Architecture

```
page.tsx (orchestration)
  ├─ AppHeader (collection selector, semantic search, theme toggle)
  ├─ DashboardPanel (resizable layout via react-resizable-panels)
  │   ├─ ScatterPlot2D (Plotly WebGL)
  │   ├─ ScatterPlot3D (Plotly 3D with spherical camera interpolation)
  │   ├─ Legend (category colors, gradient legend, click-to-mute)
  │   └─ SimilarItemsTable (semantic search results, @tanstack/react-table)
  └─ EmbeddingSidebar (controls + selected point info + text search)
```

**Collections page** (`/collections`, formerly `/test-embed`): Four tabs — HuggingFaceTab, LocalFileTab, CollectionManagerTab, SaeTab. **Tabs stay mounted once visited** (Radix `TabsContent forceMount` + `data-[state=inactive]:hidden`, visited-set gating in `page.tsx`) so in-progress form state survives tab switches; the shared `lastEmbedResult` is gated per-tab via `lastEmbedSource` (`'hf' | 'local' | 'reembed'`) on `useEmbedDataset`. **URL state**: `?tab=` + `?collection=` (manage-tab selection) round-trip via `app/collections/lib/urlState.ts` (`useCollectionsUrlState`, pure `buildCollectionsSearch`/`parseCollectionsParams`, unit-tested); the page is wrapped in `<Suspense>` for `useSearchParams`. Shared `useEmbedDataset` hook for GraphQL operations (fires sonner toasts on embed/topic completion and failure — `<Toaster/>` is mounted app-wide in `app/providers.tsx`). **Jobs & progress**: a page-global `ActiveJobsStrip` (polls `GET_EMBEDDING_JOBS` every 5s via `useEmbeddingJobs`, all statuses) lists running + interrupted jobs on every tab — so jobs stay visible after a reload — with Resume (shared `resumeJob()` dispatcher in `embeddingFormUtils.ts`, keyed on `jobType`), Cancel (hidden for `llm_labeling` — backend cancel unverified) and Remove. Client-initiated embeds/LLM-labeling show the centered blocking `ProgressModal` (a bottom-right `JobProgressDock` was tried and reverted — user judged it a downgrade: off-center, cramped at 420px); jobs whose progress is on screen are hidden from the strip via `hideJobIds`. Progress logic is layered: pure math in `lib/jobProgress.ts` (`computePercent`, `formatElapsed`, `nextEtaState` — unit-tested) → `useJobProgress(jobId)` (WebSocket subscription + elapsed + ETA, resets on jobId change) → presentational `JobProgressBody` → wrapped by `ProgressModal`. `JobsPanel`/`EmbedProgressSection` were deleted (absorbed into the strip/modal wiring in `page.tsx`). **Manage tab** is a master–detail layout: searchable `manage/CollectionListPane` (name + item/provider/topics badges; pure `filterCollections` in `lib/collectionFilter.ts`) and `manage/CollectionDetailPane` (header with Quick Action links via `buttonVariants` + `DeleteCollectionDialog` AlertDialog, preview table with click-to-expand `PreviewCell` popovers, metadata editor, SAE sections, TopicExtractionCard). **Embed flow**: numbered section cards ("1 · Data Source" … "5 · Embedding Model"), scroll-into-view on info load, and a sticky `EmbedFooterBar` (config recap from `buildEmbedSummary` + primary CTA + first blocking issue from `getEmbedValidationIssues`, both pure in `lib/embedValidation.ts` — no more `alert()`s). HF/Local share `useEmbedFormState` (columns/template/portion/model state, per-tab instances) and pure input builders (`buildHFEmbedInput`/`buildLocalEmbedInput`/`buildReEmbedInput` in `embeddingFormUtils.ts`, unit-tested); the two tabs keep intentionally distinct UIs. `PortionSelector` takes `allowedStrategies` (LocalFileTab omits `ROW_RANGE` — `EmbedLocalFileInput` has no range fields; previously shown-but-inert). TopicExtractionCard is a thin shell over `topics/TopicListSection` (rename/regenerate), `topics/ReduceTopicsSection`, `topics/LlmLabelingSection`, and `topics/TopicQualitySection`. **TopicQualitySection**: score the current topics via the `evaluateTopics` mutation (self-contained `lib/hooks/useTopicQuality.ts`, mutation-promise completion + `{collection}_evaluate` ProgressModal). Metric picker = `ToggleGroup type="multiple"` pills (Silhouette/Diversity/C_v (slow)/U_Mass; DBCV pill permanently disabled — needs the live fitted model), topic/subtopic level toggle when `hasSubtopics`. Results render as monospace badges per level with a multi-line native-`title` interpretation tooltip (pure helpers in `lib/utils/topicQualityFormat.ts`, unit-tested); persisted scores arrive via `collectionTopics.qualityMetrics` (keyed `{topic, subtopic}`, requested by GET_COLLECTION_TOPICS into `lastTopicsResult`), and fresh runs override per level (a fresh run REPLACES that level's stored blob server-side — unselected metrics drop from the badges). The jobs strip renders **below** the tab content, and jobs whose progress is already on screen (progress modal) are excluded via `hideJobIds`. Because the Card primitive is intentionally surface-less app-wide (glassy Explore overlays), the collections page root scopes a card surface for dark-mode hierarchy: `**:data-[slot=card]:bg-card/60` + soft border (`page.tsx`). **TopicConfigForm** (`app/collections/components/TopicConfigForm.tsx`, shared by TopicExtractionCard, SaeTab, and the embed-with-topics flow): exposes one merged **"Clustering Space" select** instead of separate Projection Type / Clustering Space controls (the projection toggle was ignored whenever a raw-vector mode was active). State field `clusteringSpace` holds either a stored projection (`umap_3d` default, `umap_2d`, `pca_3d`, `pca_2d` → sent as `clusterOn:"projection"` + that `projectionType`) or a raw-vector mode (`cluster_umap`/`embedding` → sent as `clusterOn`, with `projectionType` pinned to `umap_3d` for the backend's id/document load + fallback). `toTopicConfigInput` does the split back to the GraphQL pair.

## State Management

Visualization state is managed by a **Zustand store** (`lib/stores/useVisualizationStore.ts`) with `subscribeWithSelector` middleware. Components read state via selectors for granular re-renders — no prop drilling of visualization state.

- **Store shape**: `VisualizationStoreState` defines all fields; `VisualizationStoreActions` defines mutations
- **ColorScale union** (`lib/types/types.ts`): Discriminated union `{ type: 'categorical' } | { type: 'sequential'; scaleName } | { type: 'diverging'; scaleName } | { type: 'monochrome'; baseColor }`. Replaces the old flat fields (`colorScaleType`, `sequentialScaleName`, `divergingScaleName`, `monochromeColor`). `categoricalPalette` stays separate — chart components need it regardless of scale type.
- **Auto-reset subscription**: Store subscribes to `colorByField` changes and auto-clears `mutedCategories` + `hideUnclustered` (synchronous, no React effect needed)
- **Persisted preferences**: the store composes `persist` inside `subscribeWithSelector` with a `partialize` whitelist — `method`, `mode`, `nebulaMode`, `showClusterLabels`, `showAllClusterLabels`, `showAxes`, `pointOpacity`, `distanceMetric` survive reloads via localStorage key `viz-preferences`. Collection-scoped state (colorByField/colorScale/mutes/temporalRange/tooltipFields/…) is deliberately excluded — it keeps its URL-param / per-collection-default / `resetForCollectionChange` lifecycle. `skipHydration: true` + the `StoreHydrator` effect in `app/providers.tsx` rehydrate post-mount so SSR HTML and the first client render agree (no hydration mismatch; defaults flash for one frame).
- **Self-contained controls**: `ColorScaleSelector` and `VisualizationControls` read/write the store directly with zero props for state (`VisualizationControls` receives only two derived gating booleans, `hasActiveFilter`/`hasHighlights`, from DashboardPanel — topic-selection muting and highlight counts aren't in the store)
- **Data stays as props**: Points, search results, topics, and other computed data flow via props (not the store) since they come from GraphQL/hooks

## Key Patterns

- **Self-contained components**: Each manages its own loading/error/success states
- **Progressive loading**: App shell → collections → data (no full-page blocking)
- **Auto-detection** (`lib/utils/fieldAnalysis.ts`): Label fields (word, title, name, label), category fields (pos, category, type, topic), numeric vs categorical detection
- **Color system** (`lib/utils/categoryColors.ts`): Categorical (D3 + Crameri palettes), sequential (including `rainbow` via `interpolateRainbow`), diverging, monochrome scales. Presets for known types (POS, Topic). Crameri colormaps lazy-loaded from `lib/colorMaps/`. Custom color strips (hilbertColor, hueSatColor, xkcdColor) generated by backend script, stored as Crameri-format JSONs. Backend preprocessing maps hex color columns to `mapped_colour` float (0-1) for use with these strips. **Nested color mode** (`lib/utils/nestedColorUtils.ts` + `lib/hooks/useNestedCategoryData.ts`): Tableau-style hierarchical coloring where topics define base hues and subtopics get lightness variations. Activated via `nestedColorMode` toggle when `topic_label` is selected and `subtopic_label` exists.
- **Color scheme persistence** (`lib/utils/colorScaleUrl.ts`): The active colouring — field (`colorBy`), the `ColorScale` union (flattened to `scale`/`scaleName`/`color`) and `categoricalPalette` — round-trips through the URL query string (read/written in `page.tsx`), so shared links reproduce the exact look. Each collection can also store a **default colour scheme** in `extra_metadata.default_color_scheme` (JSON string, written by `SaveColorDefaultButton` via `updateCollectionMetadata`, parsed in `useCollections` → `CollectionInfo.defaultColorScheme`). On collection load `page.tsx` applies it with precedence **URL > collection default > none**; a per-collection `defaultsAppliedFor` ref plus `useEmbeddingData`'s returned `loadedCollection` gate the apply effect so it never clobbers manual edits or applies against the previous collection's stale `colorFieldOptions`. Shared helpers: `serializeColorScale`/`deserializeColorScale` (URL params) and `serializeDefaultColorScheme`/`resolveDefaultColorScheme` (per-collection default). No backend changes — relies on `extra_metadata` merge/spread.
- **Nebula cluster effects** (`lib/utils/clusterGeometry.ts` + `lib/utils/hazeRenderer.ts`): Boolean toggle (`nebulaMode`) adds translucent haze sprites around topic clusters. Uses `HazeRenderer` on a separate overlay canvas with `mix-blend-mode: screen`, hooked into Plotly's GL render loop for camera-synced projection. Deprecated renderers (volume isosurface, WebGL sprites, Three.js bloom) archived in `lib/utils/experimental/`. **Per-frame cost is minimized** in `ScatterPlot3D`: the renderer lifecycle (create/dispose + `glplot.onrender` hook, keyed on `nebulaMode`/`plotReady`/`isDark`/has-clusters) is split from cluster-content updates, so muting/filtering/recoloring only calls `updateClusters` on the existing renderer instead of re-instantiating the Three.js `WebGLRenderer`; the overlay-canvas sync writes styles + backing-store dims only on an actual offset/size change (`lib/utils/overlayLayout.ts` — sharing that math with the label overlay, avoiding a per-frame `canvas.width` reset + reflow); and the data→scene model matrix is cached per `bounds` identity (pinned axis ranges make it constant during camera interaction). Each haze sprite is still its own draw call — batching to `THREE.Points` (one draw call per cluster) is the documented Tier 2 follow-up. `clusterDataMap` excludes points muted by text-search/temporal filters (`combinedMutedIndices`) in addition to `mutedCategories`/`hideUnclustered`, so haze and cluster labels track visibly active points; `HazeRenderer.dispose()` clears the drawing buffer to the neutral blend color so no stale haze frame lingers on the still-mounted overlay canvas when filtering empties all clusters.
- **Label collision avoidance** (`app/utils/labelPlacement.ts` + `lib/utils/collisionGrid.ts`): On each rAF-polled camera change, projects 3D label positions to 2D screen space via MVP matrices from Plotly's WebGL internals, measures text bounding boxes on a canvas overlay, then uses `CollisionGrid` with two-pass insertion (thin probe + full box) for greedy set packing. Higher-similarity labels survive collisions; selected point gets infinite priority. **Cluster labels** (`showClusterLabels` on `VisualizationState`): displays topic/subtopic names at cluster centroids (bold 13px, cluster color) with highest collision priority, sharing the same `CollisionGrid` as point labels.
- **Resizable Legend** (`lib/hooks/useVerticalResize.ts`): Legend card has a drag handle at its bottom edge controlling `maxHeight` via mouse/touch events. Dragging below the collapse threshold (60px) collapses to a pill button (Palette icon); clicking the pill re-expands at default height. Uses callback ref pattern for conditionally rendered elements.
- **Screenshot export** (`lib/utils/plotCapture.ts`): A camera modebar button (rightmost, both plots) downloads one PNG of the current visualization — plot + overlay canvases + legend, no UI — at screen size × devicePixelRatio. Background is filled explicitly (pure black in dark mode, white in light) because the plot/paper backgrounds are transparent in dark mode. 2D captures via `Plotly.toImage` (scattergl axes are SVG, a live-canvas copy would miss them) then draws the label canvas on top; 3D reads the live GL scene via the fork's `scene._scene.toImage()` (synchronous redraw + readPixels — immune to `preserveDrawingBuffer:false` and to the fly-to camera lagging the layout) and composites the haze canvas with `globalCompositeOperation` matching its CSS `mix-blend-mode`, then labels. The plots' Plotly `config` memos only rebuild on plotly-ready, so the button's click handler goes through a `screenshotHandlerRef` reassigned every render (same anti-staleness pattern as `renderLabelsRef`). Each plot composites what it owns and calls its `onScreenshot(canvas)` prop; `DashboardPanel` then rasterizes the **live legend DOM node** with `html-to-image` (`drawLegendOverlay` — the real component, histogram and all; skipped when legend hidden/collapsed since the overlay ref is unmounted; elements marked `data-export-exclude`, e.g. the drag handle, are omitted) over a canvas-blurred backdrop replicating the Card's `backdrop-blur-sm` (foreignObject rasterization can't reproduce backdrop-filter), and downloads via `downloadCanvasPng` (`downloadJson.ts`).
- **Responsive plots**: `useContainerDimensions` (ResizeObserver) + Plotly `uirevision: 'true'` preserves zoom/pan
- **Controls panel layout** (`app/components/VisualizationControls.tsx`): grouped into uppercase sections — **View** (Projection `[PCA|UMAP|Manual]` and Dimensions `[2D|3D]` as two labeled segmented `ToggleGroup` rows; Manual adds a compact X/Y/Z input grid + "Show axis lines"; Nebula is 3D-only), **Color** (field select + two `circularghost` icon buttons: `ColorScaleSelector` palette and overwrite-only `SaveColorDefaultButton`; a saved default is removable via the Manage-tab metadata editor), **Labels** ("Label search results" gated on `hasHighlights`; cluster-label toggles gated on `colorByField`; whole group hidden when empty), **Points** (point opacity always; "Show only highlighted" gated on `hasHighlights`; "Hide filtered points" + "Filtered point opacity" gated on `hasActiveFilter`), **Tooltip Fields**. The gating booleans come from `DashboardPanel` (via `EmbeddingSidebar`) because `combinedMutedIndices`/`effectiveMutedCategories`/`highlightedCount` are computed there, not in the store; a `DashboardPanel` effect clears `showOnlyHighlighted` when highlights disappear so the hidden checkbox can't strand an empty plot (`hideFilteredPoints` deliberately persists). "Show only highlighted"/"Show labels" moved here from SearchSidebar; Distance Metric moved the other way — Search panel owns *what to find*, Controls owns *how to draw*.
- **Search flow**: Text search is **server-side** via the `textSearch` GraphQL query (`lib/hooks/useTextSearch.ts`). Backed by DuckDB `ILIKE` substring scans on the per-dataset items table (`json_extract_string` for metadata fields) — see `duckdb_client.text_search()`. `SearchSidebar` Advanced section has field selector (multi-select: Document + metadata fields), match mode (Contains/Exact), case-sensitivity toggle, metadata Filters, Query Prompt Name, and the semantic-search Distance Metric select (moved from the Controls panel; reads/writes the store directly). Default: document-only, contains, case-insensitive. `TextSearchConfig` state stored in Zustand (`useVisualizationStore`). Text search uses **muting** (not glow) — non-matching points turn gray/low-opacity while matches keep normal colors. Text search muting (`searchMutedIndices`) combines with temporal muting (`temporallyMutedIndices`) into `combinedMutedIndices` in both scatter plots. Clicking a text search result in the sidebar triggers semantic search via GraphQL → similar items get glow overlay on top of muting. `HighlightedText` component marks matching text in both labels and documents in search results. **Topic search** (`lib/hooks/useTopicSearch.ts` + `app/components/TopicSearchSection.tsx`): Browse/filter topics by name/keywords, or rank topics by semantic similarity. Selecting topics highlights their points and scopes semantic search via backend `filters` parameter.
- **Synchronized topic selection**: When `colorByField === 'topic_label'`, `selectedTopicIds` (from `useTopicSearch`) is the single source of truth. `DashboardPanel` derives `effectiveMutedCategories` via `useMemo` (no state writes, single render cycle). Legend clicks, TopicSearchSection toggles, and 3D cluster label clicks all call `toggleTopic()`. Non-topic color fields preserve existing `mutedCategories` behavior. 3D cluster labels are clickable via `onClickCapture` hit-testing against bounding boxes stored during `renderLabels()`.
- **Filtered point controls**: `hideFilteredPoints` (boolean) on `VisualizationState` removes muted points entirely from the plot. `mutedPointOpacity` (0–1, default 0.20) is a **multiplier** applied to the current base opacity (not an absolute value) — muted opacity = base opacity × multiplier. This scales properly with point count since base opacity already decreases for large datasets. Toggle + slider UI in VisualizationControls. Reset on collection change.
- **Zoom-out limit** (`lib/hooks/useZoomLimit.ts`): Shared hook intercepting wheel events in capture phase before Plotly processes them. 3D reads live camera distance from `glplot.camera.eye` (max 3.0); 2D reads axis ranges from `_fullLayout` (max 2x data extent). Zoom-in, pan, rotation unaffected.
- **Deferred selected point** (3D only): On point click, `selectedPoint` drives camera fly-to via direct glplot manipulation (no Plotly.react). Trace-visible `renderedSelectedPoint` only syncs when `highlightedIndices` (search results) arrives, keeping `plotData` stable during animation to avoid main-thread stalls on large datasets.
- **Analytics category list** (`app/components/charts/CategoryBarList.tsx` + `lib/utils/categoryRowData.ts`): Category distribution as horizontal background-fill rows — a translucent category-color bar (inset 2px vertically, square baseline left, 4px-rounded data end) fills each row (width ∝ count, normalized to the max visible category, floored to an 8px chip so tiny categories don't look clipped) with label + count (`count (pct%)`) on top. All categories render in a scrollable list (shadcn `ScrollArea`, 224px viewport, no top-N cap). **Categorical-only guard**: `AnalyticsSidebar` only feeds the list fields whose `recommendedScale === 'categorical'` (cardinality fallback ≤300 for fields missing from `colorFieldOptions`) — continuous color fields (sequential/diverging) would otherwise render one row per distinct float; they fall back to the field-picker mode, and the temporal cross-tab is skipped for them too. Independent field selector dropdown (categorical fields from `colorFieldOptions`), defaulting to the active color field. When a text-search or temporal filter is active (`filteredCounts` prop, computed from `combinedMutedIndices`), rows show a faded total track + solid matching fill and counts flip to "matches / total". Sort ToggleGroup: Count (default; matches-desc while filtered) / Rate (match rate, only offered while filtered) / Name (numeric-aware natural order via `sortPeriods`). Category name filter input appears when >15 categories, matching resolved display labels. Row/sort/summary logic is pure and unit-tested (`lib/utils/__tests__/categoryRowData.test.ts`). Click row = isolate, shift+click = toggle (same `onCategoryToggle` contract). Unclustered noise (HDBSCAN label -1) is excluded from the distribution for **both topic and subtopic** fields via `getUnclusteredValues()` (`categoryColors.ts`, noise tokens derived from `TOPIC_PRESET`) — chart-only, independent of the scatter `hideUnclustered` toggle. The superseded recharts `CategoryBarChart.tsx` is retained temporarily pending verification of the swap.
- **Temporal filtering** (`lib/utils/temporalAnalysis.ts` + `lib/hooks/useTemporalData.ts` + `app/components/charts/TemporalFilterChart.tsx`): Auto-detects year/date fields via name heuristics and numeric range analysis. Analytics sidebar shows temporal distribution chart even without a color field selected (standalone mode: simple count-per-period area chart; stacked mode: category breakdown when color field is set). Custom DOM overlay range picker with draggable handle bars and a slidable selected region lets users filter points by time period — points outside the selected range are muted (gray, low opacity) in both 2D and 3D scatter plots via trace-splitting. Double-click resets to full range. `startTransition` + `useDeferredValue` keeps interaction non-blocking. Temporal range persists across color field changes but clears on collection change. Syncs with topic muting (muted categories excluded from stacked chart).
- **Direction probes** (`app/components/ProbeSection.tsx` + `lib/hooks/useProbes.ts` + `lib/utils/probeFields.ts`): "Direction Probes" block in the Analytics sidebar — pick a numeric metadata field (defaults to the active color field; probe-derived fields excluded) + kind (Ridge/Mass-mean/SVR-RBF/Logistic/MLP — logistic is disabled unless the target is binary, i.e. `uniqueCount === 2`, and its badge shows AUC/accuracy instead of R²/ρ), fit server-side via the `trainProbe` mutation. A **gear popover** (`ProbeSettingsPopover`) next to the kind select exposes per-kind hyperparameters (alpha / C + kernel / class weight / hidden size + epochs) plus shared seed + train split; the pure model lives in `lib/utils/probeParams.ts` (`ProbeParams`, `probeParamFields`, `buildTrainProbeInput` — only non-default values are sent so the backend's defaults stay authoritative; unit-tested) (inline progress from the shared job subscription, jobId `{collection}_probe`; subscription is display-only — completion keys off the mutation promise). `useProbes` owns the probe list + per-probe score arrays (`GET_PROBE_SCORES`, **no-cache** — retrains reuse identical args and 150k-float arrays don't belong in Apollo's cache) and produces `augmentedData`: a fresh `data` object whose `itemMetadata` has `probe_<field>_<kind>_score`/`_residual` keys merged in (pure helpers in `probeFields.ts`, unit-tested). `page.tsx` feeds `augmentedData ?? data` ONLY into `useVisualizationPoints` and appends `probes.fieldOptions` to `colorFieldOptions` — the persisted `field_analysis` cache and SearchSidebar's server-side metadata filters never see probe fields. Auto-recolor to the score field fires after scores land (pendingRecolor ref); score button → sequential scale, residual button → diverging (applied at click time — `ColorFieldOption.recommendedScale` can't express diverging); deleting the active probe clears `colorByField` first. The target-field select stays controlled (`value={effectiveField ?? ''}`) and its follow-the-color-field logic (`resolveProbeTargetSelection`, unit-tested) **pins the just-probed field** when training auto-recolors to a `probe_*` field — otherwise, in follow mode (no explicit pick), the followed field would resolve to nothing after fitting and disable the Fit button, blocking a second kind on the same field. **Target eligibility** (`isProbeTargetOption`): numeric fields plus **binary categorical** fields (exactly two distinct values, trained server-side as 0/1). For a binary probe the badge's native tooltip appends the class mapping (`formatTargetMapping`, e.g. "safe → 0 · unsafe → 1") read from `ProbeInfo.targetMapping`. Badge shows val R² + ρ (massmean's R² is a backend-calibrated readout); full metric list in the tooltip.

## Troubleshooting

- **Plotly types**: Use type assertions (`as Partial<Config>`)
- **Color scale not loading / custom strip shows as viridis**: Crameri/strip colormaps (e.g. `hilbertColor`) are lazy-loaded JSON chunks read synchronously from cache via `getCrameriPlotlyScale()`; an unloaded strip falls back to viridis. The scatter plots call `useColorScaleReady(colorScale)` (`lib/hooks/useColorScaleReady.ts`) to lazy-load **the active** strip and bump a tick that forces the colorscale `useMemo` to recompute when it lands — needed because a scale set programmatically (URL param / collection default) never opens `ColorScaleSelector` (which is the only other thing that triggers the load). Include that tick in any new colorscale memo deps.
- **Categories all gray**: Check `buildCategoryColorMap()` — presets only override specific values
- **Sidebar/ScrollArea content flush against the right edge (right padding clipped)**: Radix ScrollArea wraps viewport children in a `display: table; min-width: 100%` div, which grows *past* the viewport when any child has a large intrinsic width (a measured recharts svg, nowrap `truncate` labels) — the overflow is clipped (`overflow-x: hidden`), swallowing the content's right padding. Since the recharts svg re-measures against the wider wrapper, the state is self-sustaining. Fix: force the wrapper to block with `[&>[data-radix-scroll-area-viewport]>div]:block!` on the ScrollArea — applied in the `SidebarContent` primitive (`lib/ui-primitives/sidebar.tsx`) and CategoryBarList's inner list. Only do this where horizontal scrolling is never wanted (tables that scroll horizontally need the table wrapper).
- **`scene.scatter2d.update is not a function` / 2D scattergl renders nothing (3D fine)**: two related causes. (1) plotly marks regl components with the placeholder `true` during the calc pass (`calc.js`) and only creates the real objects in the dirty plot call (`plot.js`); a pan/zoom drag or box-select landing in that window called `.update()` on the boolean. Patched in the fork (`forked/plotly.js/src/traces/scattergl/scene_update.js`, `select.js`, `edit_style.js`) with `isReady()` guards. (2) The regl-* packages call `glslify()` **at runtime** around their pre-compiled shader strings, so `next.config.ts` must NOT alias glslify to `false` — that makes `createScatter()` throw (silently swallowed by react-plotly, which has no `onError`), leaving `scene.scatter2d` stuck on the placeholder → blank plot. The alias points to `glslify-runtime-stub.js` (tagged-template join) instead. **Under Next 16 this alias must ALSO be declared in `turbopack.resolveAlias` (Turbopack ignores the `webpack()` function) — see "Build & bundler".** Note the breakage only surfaces after a clean `.next` rebuild — stale webpack-cached chunks can mask config-level regressions for a long time. Also: never run two `next dev` instances against the same `.next` (corrupts chunks → ENOENT / 404s / silent server death).
