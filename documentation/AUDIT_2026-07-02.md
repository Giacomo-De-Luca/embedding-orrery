## Backend — data layer (DuckDB/ChromaDB)

### [HIGH] Multi-table writes have no transactions; the _exec cursor-per-call design makes atomicity impossible
`interpretability_backend/backend/clients/duckdb_client.py:543` — bug

All claims verified with exact line matches, except one adverb: a crash mid-delete_dataset leaves the DB half-deleted until manually retried, not permanently — the `datasets` row is deleted last and the DELETE/DROP-IF-EXISTS statements are idempotent, so re-invoking delete_dataset completes the cleanup. Nothing detects or prompts that retry, and the inconsistent state (e.g., topics gone, dataset and items table still served) persists and is visible in the meantime. The create_topic_extraction hazards are as stated: a crash between the UPDATE (lines 1235-1241) and INSERT (1244-1250) leaves no active extraction, and with no unique constraint on (collection_name, is_active), concurrent calls can leave two active rows that get_active_topics (line 1308, LIMIT 1 without ORDER BY) resolves arbitrarily. The proposed fix direction (transaction context manager pinning a single cursor or using self._conn under _write_lock) is sound, since per-call cursors from _exec (line 105) each have independent transaction state.

### [MEDIUM] search_documents_by_activations builds SQL VALUES by f-string interpolation — NaN/Inf activations crash the query; only unparameterized value path in the file
`interpretability_backend/backend/clients/duckdb_client.py:2001` — bug

The unparameterized f-string VALUES interpolation at duckdb_client.py:2001 is real, and DuckDB 1.4.1 rejects bare non-finite literals exactly as claimed (`Binder Error: Referenced column "nan" not found in FROM clause!` — same for `inf`). However, via the cited production path (mutations.py:1183 → run_prompt_highlight), NaN activations cannot reach the SQL: _max_pool_activations filters with `max_pooled > 0`, and NaN > 0 is False in torch, so NaNs are dropped upstream. Only +Inf (which passes the > 0 mask, e.g. from fp16/MPS overflow) can crash the query through that path; NaN triggers the crash only for direct callers of the public search_documents_by_activations method (e.g. other future call sites). The recommended fix (filter non-finite values and bind parameters) remains correct.

### [LOW] FTS index lifecycle is broken: delete_dataset orphans the fts_main_* schema, and the in-memory dirty flag loses pending rebuilds across restarts

### [LOW] Lazy singleton is not thread-safe — first-call race can create two DuckDBClient connections with split write locks
`interpretability_backend/backend/API/duckdb_instance.py:16` — bug

get_duckdb_client uses unsynchronized check-then-create. The backend routinely calls it from multiple threads: embedding mutations run via asyncio.to_thread (duckdb_sync._get_db) while GraphQL queries resolve concurrently, and nothing in main.py's lifespan pre-initializes the client (it only calls ensure_seed_loaded). If two threads race on first call, each constructs a DuckDBClient — meaning two duckdb.connect() handles, double _ensure_schema execution, and crucially two independent `_write_lock`s, so the lock that is supposed to serialize the DataFrame replacement-scan writes (duckdb_client.py:84) no longer guards anything: concurrent INSERT ... SELECT FROM df on the same table can then hit DuckDB write-write TransactionExceptions. chromadb_instance.py:16 has the identical pattern. Fix: guard creation with a module-level threading.Lock (double-checked), or instantiate both singletons eagerly in the FastAPI lifespan.

### [LOW] Text search treats %/_ in the user query as SQL wildcards, and case-sensitive matches get case-insensitive (or missing) snippets
`interpretability_backend/backend/clients/duckdb_client.py:1133` — bug

The contains-mode predicates concatenate the raw query into LIKE/ILIKE patterns without escaping, so a user searching for "100%" matches any document containing "100" followed by anything, and "_" matches any single character — surprising matches whose snippet then comes back None because the Python-side literal find (`doc.lower().find(query_lower)`, line 1141) can't locate the pattern text. Separately, in case_sensitive mode the snippet position is still computed with the lowercased find, so the snippet can be extracted around a different-case occurrence than the one SQL actually matched. Fix: escape %/_/\\ in the query (LIKE ... ESCAPE '\\') and use `doc.find(query)` when case_sensitive.

## Backend — GraphQL queries

### [HIGH] Filtered semantic search silently drops matches: hardcoded 100k pre-filter cap plus 5x over-fetch heuristic
`interpretability_backend/backend/API/queries.py:532` — bug

Both defects exist, but the 100k-cap mechanics are misstated. Corrected: get_filtered_items applies 'ORDER BY row_index LIMIT 100000' AFTER the metadata WHERE clause (duckdb_client.py:746), so truncation occurs only when a filter matches more than 100,000 rows — in that case matching items beyond the first 100k matches (in row_index order) are silently excluded from allowed_ids, e.g. a broad temporal range or multi-topic filter on the 153k-row WordNet dataset. Items with row_index >= 100000 are NOT unconditionally excluded; a filter matching fewer than 100k rows returns all matches regardless of row_index. The second defect is confirmed as stated: fetch_n = n_results * 5 (queries.py:539, 637) with no iterative re-fetch means a selective filter (e.g. narrow temporal range sent by useAppSearch.ts as FilterInput[], n_results=20 → fetch 100) can leave fewer than n_results survivors after the allowed_ids post-filter, silently returning a short or empty result list even though more matching neighbors exist in ChromaDB. This selective-filter case is the more commonly triggered of the two.

### [HIGH] Every query resolver is sync and does blocking I/O on the event loop, stalling WebSocket token/progress streams
`interpretability_backend/backend/API/queries.py:158` — performance

All Query resolvers are plain 'def' methods. Strawberry 0.285.0 executes sync resolvers inline inside the async execution (verified: no to_thread/run_in_threadpool/sync_to_async in the installed strawberry/schema/schema_converter.py), so they block uvicorn's event loop. Worst offenders: huggingface_dataset_info/huggingface_dataset_preview do network I/O (huggingface_client.py:126 hf_dataset_info(), :148 load_dataset_builder(), :227 load_dataset(streaming=True)); collection() loads and JSON-parses the entire items table (153k rows for WordNet); semantic_search with a text query triggers a 100MB+ embedding-model load on first call; local_file_info/preview do pandas disk I/O. While any of these run, the generateStream token subscription and embedding-progress WebSocket subscriptions on the same loop stall completely. Mutations already use asyncio.to_thread (per backend CLAUDE.md); queries should be 'async def' wrapping the blocking work in asyncio.to_thread the same way.

### [MEDIUM] collection() assumes all projection types align positionally with the first type's ids — partial projections silently misalign
`interpretability_backend/backend/API/queries.py:303` — bug

collection() (queries.py:300-305) has no length or id-alignment check between items_data['ids'] (from the first projection type with data) and each other type's coordinates array — confirmed. But the claimed trigger is wrong: insert_projections_batch does NOT write incrementally; it is one atomic INSERT OR REPLACE per projection type over the full id list, so an interrupted job leaves whole types missing (handled as null), never a partially-written type. The real trigger is cross-run divergence: when a collection's item set grows and projections are recomputed, a crash between the per-type inserts (e.g., killed during the UMAP fit after the PCA inserts) leaves fresh PCA types with N+M rows and stale UMAP types with N rows keyed to the old item subset. The misaligned state persists because a resume that embeds 0 new items skips projection recomputation (embedding_pipeline.py:86 gates on total_embedded > 0). Frontend impact (useVisualizationPoints.ts positional indexing) then depends on row_index layout: interleaved stale subsets (random-sample portions) render points at the wrong item's coordinates after the first gap; prefix subsets (first-N appends) merely truncate the newer items. The suggested fix (verify len(coordinates) == len(ids), ideally id order, and null out mismatched types) is appropriate.

### [HIGH] Timeout paths release the GPU lock while the inference thread is still running (asyncio.wait_for cannot cancel to_thread)
`interpretability_backend/backend/API/mutations.py:726` — bug

The core defect is real: in five resolvers (mutations.py load_model 689-693, run_prompt_activations 728-743, generate_steered_response 769-779, run_prompt_highlight 808-818, search_documents_by_prompt 1158-1168) a wait_for timeout cancels only the awaiting task; the to_thread worker keeps running GPU inference, and the `async with service._lock` exits during unwinding, releasing the lock mid-forward. The mutation-path service methods accept no cancel event, so the orphan cannot be stopped, and a subsequent retry or unloadModel runs concurrently with it. Two corrections: (1) subscriptions.py is NOT a correct contrast on lock ordering — its `await task` at lines 176-180 is in the except handler, which runs after the `async with` has already released the lock (the in-file comment is wrong); what actually protects the subscription path is the per-token cancel_event that halts the thread promptly (plus the in-lock `await task` at line 171 on the normal path). (2) The unload consequence is overstated: torch.mps.empty_cache() frees only unused cached blocks, and the orphan holds its own wrapper reference, so `_wrapper = None` won't free the model mid-forward. The realistic hazards are concurrent GPU forward passes (defeating the lock), a race on the lazily-rebuilt self._prompt_explorer (interpret_service.py:429-451, nulled at 296), and double model residency (reload after unload while the orphan still runs) causing MPS memory exhaustion. The proposed fix (cancel event checked in the service, or awaiting the thread before lock release) is sound.

### [MEDIUM] Permutation test can report empirical_p = 0.0, which is statistically invalid
`interpretability_backend/interpret/utils/mantel.py:120` — bug

empirical_p is computed as the plain fraction of null rhos >= observed. When no permutation exceeds the observed rho (the common case for a real signal with n_perms=1000) this reports p = 0.0, but a Monte-Carlo p-value can never be 0 — the standard correction (North et al. 2002; also used by scikit-bio's Mantel implementation) is (count + 1) / (n_perms + 1), giving p <= 1/1001 rather than an impossible exact zero. Fix: `empirical_p = (float(np.sum(null_rhos >= observed_rho)) + 1) / (n_perms + 1)`.
